# Бюджет исполнения и статистика

Два связанных механизма в `ScriptEngine`:

- **Бюджет** — не даёт скриптам (особенно кривым модам) ронять FPS: суммарная
  работа за кадр ограничена, лишнее переносится на следующий кадр.
- **Статистика** — окно наружу: сколько файберов живо, что делает каждый
  триггер, кто жрёт бюджет. Поверх неё позже строится редакторское окно.

Оба почти бесплатны в горячем пути: счётчик инструкций тика нужен для бюджета
всё равно, а детальная разбивка — за флагом.

---

## Бюджет

Единица — **инструкция байткода**. В начале каждого `Tick(dt)` (≈ кадр) бюджет
сбрасывается; вся работа скриптов за кадр (включая события, поднятые после
`Tick` в том же кадре) черпает из него. Кончился бюджет — недоделанные файберы
остаются готовыми и продолжат в следующем кадре. Их состояние (стек, кадры,
позиция) сохраняется — ничего не теряется и не перезапускается.

### Настройки (`ScriptEngine`)

| Поле | По умолчанию | За что отвечает |
|------|--------------|-----------------|
| `TickInstructionBudget` | 5 000 000 | Потолок инструкций на один `Tick` (кадр). Дренаж очереди останавливается при исчерпании. |
| `MaxInstructionsPerFiberRun` | 200 000 | Слайс одного запуска файбера: не даёт одному файберу выесть весь бюджет разом, обеспечивает честный round-robin. |
| `MaxFibersStartedPerTick` | 4096 | Потолок НОВЫХ файберов (`spawn` + `ActivateTrigger`) за тик. Защита от спавн-бомбы: сверх лимита `spawn` тихо отбрасывается (с одним предупреждением за тик). Обработчики событий сюда не считаются — их число ограничено числом триггеров. |
| `Policy` | `CarryOverThenKill` | Что делать с файбером, упёршимся в бюджет (см. ниже). |
| `BudgetKillThreshold` | 300 | Для `CarryOverThenKill`: сколько тиков ПОДРЯД у бюджета до убийства. |
| `EnableProfiling` | `false` | Детальная разбивка инструкций по триггерам (см. «Стоимость»). |

### Политики (`BudgetPolicy`)

- **`CarryOver`** — только переносить в следующий тик. Логика никогда не рвётся,
  но настоящий вечный цикл (`while(true)` без `wait`) будет жрать бюджет каждый
  кадр вечно. Годится, если модам полностью доверяете.
- **`CarryOverThenKill`** (по умолчанию) — переносить, но если файбер упирается
  в бюджет `BudgetKillThreshold` тиков ПОДРЯД, не делая `wait` — убить с ошибкой
  в лог. Живая логика (тяжёлый, но конечный расчёт, разбитый на кадры) цела;
  бесконечный цикл ловится. Счётчик подряд-исчерпаний обнуляется, как только
  файбер кооперативно приостанавливается (`wait`/`wait until`) или завершается.
- **`KillImmediately`** — убивать при первом же исчерпании. Жёстко, зато
  нагрузка предсказуема.

### Событие

`event Action<int> OnBudgetExceeded` — поднимается, когда файбер УБИТ политикой
бюджета; аргумент — id триггера-владельца (или -1). Редакторское окно по нему
подсветит виновника. Само по себе исчерпание бюджета (перенос) события не
поднимает — только убийство.

### Важное про кадры

Бюджет сбрасывается в `Tick`. Адаптер Unity зовёт `Tick` каждый `Update`, так
что «бюджет на тик» = «бюджет на кадр». Если игра поднимает события, но не зовёт
`Tick`, бюджет не восстановится и всё встанет в очередь — держите `Tick` в
игровом цикле. Перенос обработчика на следующий кадр под нагрузкой слегка
меняет W3-семантику «немедленно» — это осознанный компромисс на случай перегруза
(случаться должен редко).

## Статистика

Два уровня: глобальный снимок (`EngineStats`) и пер-триггерный
(`TriggerStats`).

### `EngineStats GetStats()`

Value-тип, без аллокаций. Датчики считаются одним проходом по пулу файберов.

**Мгновенные датчики:**
- `LiveFibers` — всего живых файберов.
- `RunningFibers` — исполняются прямо сейчас (при чтении между кадрами ≈ 0).
- `SleepingFibers` — спят по таймеру (`wait N`).
- `YieldedFibers` — ждут следующего тика (`wait until`).
- `ReadyFibers` — в очереди на запуск (в т.ч. перенесённые бюджетом).
- `DynamicStrings` — размер динамического сегмента `StringTable`.
- `Collections` — сколько живых коллекций (массивы + списки + мапы).

**Накоплено с прошлого `Tick`** (≈ за кадр, обнуляется каждым `Tick`):
- `InstructionsThisTick` — исполнено инструкций.
- `FibersStartedThisTick` / `FibersCompletedThisTick` — запущено / завершилось.
- `FibersKilledThisTick` — убито политикой бюджета.
- `FibersErroredThisTick` — умерло от скриптовой ошибки.
- `EventsRaisedThisTick` / `HandlerInvocationsThisTick` — поднято событий /
  запущено обработчиков.
- `FibersDeferredThisTick` — перенесено в следующий тик из-за бюджета.
- `SpawnsDroppedThisTick` — спавнов отклонено лимитом (спавн-бомба).
- `BudgetExhaustedThisTick` — упёрлись ли в `TickInstructionBudget`.
- `Time` — модельное время движка.

### `GetTriggerStats(List<TriggerStats> into)`

Заполняет ПЕРЕДАННЫЙ список (переиспользуйте его — тогда без аллокаций каждый
кадр). Есть и удобная перегрузка `List<TriggerStats> GetTriggerStats()`, которая
аллоцирует новый список — для разовых вызовов.

Поля `TriggerStats`:
- `TriggerId`, `Name`, `Module`, `Enabled` — кто это и включён ли.
- `TimesFired` — сколько раз запускались его обработчики (кумулятивно).
- `FibersSpawned` — сколько файберов породил (обработчики + `spawn` + `activate`).
- `FibersAlive` — сколько его файберов живо сейчас.
- `LastFiredTime` — когда сработал в последний раз (модельное время).
- `ErrorCount` / `LastError` — сколько его файберов упало и текст последней ошибки.
- `InstructionsThisTick` / `InstructionsTotal` — **только при `EnableProfiling`**,
  иначе 0.

Это ровно то, что дизайнер спрашивает первым: «сработал ли мой триггер, сколько
раз, живы ли его файберы, не упал ли».

## Стоимость и профайлинг

- Общий счётчик инструкций тика (`InstructionsThisTick`, бюджет) инкрементится
  всегда — он нужен для самого бюджета, так что это «бесплатно».
- Простые счётчики (события, старты/смерти файберов, срабатывания триггеров) —
  пара инкрементов на действие, ничтожно.
- Датчики (`LiveFibers` и пр.) считаются только в момент вызова `GetStats` /
  `GetTriggerStats` — проходом по пулу. Зовите их с частотой окна отладки
  (напр. раз в 0.25 с), а не каждый кадр в проде.
- **`EnableProfiling`** добавляет разбивку инструкций по триггерам — это запись
  в горячем пути после каждого запуска файбера. Держите **off в релизе** и
  **on в редакторе**. Без него `InstructionsThisTick/Total` в `TriggerStats`
  остаются нулями, всё остальное работает.

### Рекомендации

- **Релиз:** `Policy = CarryOverThenKill`, `EnableProfiling = false`, бюджет под
  свой бюджет кадра (5M — грубый старт; калибруйте профайлером под целевое
  железо).
- **Редактор/отладка:** `EnableProfiling = true`, при желании понизьте
  `TickInstructionBudget`, чтобы раньше видеть, кто зарывается.

## Пример

```csharp
var engine = new ScriptEngine(registry);
engine.Policy = BudgetPolicy.CarryOverThenKill;
engine.OnBudgetExceeded += triggerId =>
    Debug.LogWarning($"Триггер {triggerId} убит бюджетом — вероятно, цикл без wait.");

#if UNITY_EDITOR
engine.EnableProfiling = true;
#endif

// ...в окне отладки (не каждый кадр):
var stats = engine.GetStats();
Debug.Log($"файберов: {stats.LiveFibers}, инстр/кадр: {stats.InstructionsThisTick}");

_triggerBuffer ??= new List<TriggerStats>();
engine.GetTriggerStats(_triggerBuffer);
foreach (var t in _triggerBuffer)
    if (t.ErrorCount > 0)
        Debug.LogWarning($"{t.Name}: ошибок {t.ErrorCount} — {t.LastError}");
```

Редакторское окно (список живых файберов, кто на каком `wait`, кто жрёт бюджет)
строится поверх этих двух методов и события `OnBudgetExceeded` — отдельным
шагом, ядру для этого меняться не нужно.
