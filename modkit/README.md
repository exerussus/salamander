# Модкит — среда для написания модов

Самодостаточная папка: открой её в VS Code и пиши моды. Ничего из движка игры
или Unity-проекта здесь не нужно — только редактор, расширение и чекер.

## Что внутри

```
modkit/
  .vscode/               настройки уже указывают на чекер и папку модов
  bin/                   сюда кладётся DslCheck (см. bin/README.md)
  mods/
    salamander-api.json      API игры: что доступно модам (даёт разработчик игры)
    example_mod/         рабочий пример мода
      module.json
      src/hello.sal
```

## Первый запуск (3 шага)

1. **Расширение.** Установите «Salamander» в VS Code
   (VSIX от разработчика игры или из исходников `Tools/vscode-salamander`).
   При открытии этой папки VS Code сам предложит его установить.
2. **Чекер.** Положите `DslCheck` в `bin/` (см. `bin/README.md`).
3. **Откройте папку** `modkit` в VS Code (File → Open Folder).

Готово. Откройте `mods/example_mod/src/hello.sal` — работает подсветка,
автодополнение по API игры и проверка ошибок при сохранении.

## Как писать мод

1. Скопируйте `mods/example_mod` в `mods/<имя_вашего_мода>`.
2. Впишите имя в `module.json` (`name`) и перечислите файлы в `sources`.
3. Пишите триггеры в `src/*.sal`. Подсказки:
   - наберите `event ` внутри триггера — выпадут события игры с сигнатурами;
   - `Engine.` — встроенные методы движка;
   - `UnitApi.` / `BattleApi.` (и любой класс API) — методы игры;
   - `Team.` / любой енум — его элементы;
   - ошибки компиляции подчёркиваются прямо в коде (панель Problems).
4. Триггер обязан содержать хотя бы один `event` — иначе это `class`.

## Проверить без игры

Сохранение файла запускает проверку автоматически. Вручную — палитра команд
(**Ctrl+Shift+P**) → «Salamander: проверить скрипты». Или из терминала:

```bash
# .dll (нужен .NET)
dotnet bin/DslCheck.dll mods --api mods/salamander-api.json
# самодостаточный бинарник
./bin/DslCheck mods --api mods/salamander-api.json
```

Код возврата 0 — ошибок нет, 1 — есть (удобно для проверок перед публикацией).

## Поставить мод в игру

Скопируйте папку вашего мода (`<имя_вашего_мода>` целиком, с `module.json`)
в папку модов игры. Точный путь укажет разработчик игры — обычно это
`.../<Игра>_Data/StreamingAssets/Scripts/` или пользовательская папка модов.
Игра подхватит мод при запуске; если включён хот-релоад — можно править и
видеть изменения на лету.

## Режим исполнения (`execution` в module.json)

- `"cooperative"` (по умолчанию) — есть файберы: можно `wait N`, `wait until`,
  `spawn` для параллельной и отложенной логики (стратегии, RPG).
- `"synchronous"` — без файберов: обработчик события исполняется целиком до
  конца, `wait`/`wait until`/`spawn` запрещены компилятором (карточные,
  пошаговые игры — всё предсказуемо и по порядку). Синхронный модуль может
  зависеть только от синхронных.

Игра может писаться под конкретный режим — уточните у разработчика игры, в
каком режиме пишутся моды для неё.

## Если API игры обновился

Разработчик пришлёт новый `salamander-api.json` — замените им файл в `mods/`.
Автодополнение и проверка сразу подхватят новые события и методы.
`apiVersion` в вашем `module.json` должен совпадать с версией в манифесте;
иначе чекер (и игра) сообщат о несовместимости.
