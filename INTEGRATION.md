# Интеграция в игру и подготовка среды для модеров

Главный принцип разделения: **в Unity попадает только то, что исполняется в
игре**; инструменты (чекер, расширение, демо-хосты) живут отдельно и в билд
не входят. Разделение достигается тем, ЧТО вы копируете в `Assets`, — править
рантайм для этого не нужно.

## Три зоны

| Зона | Что | Куда | В билде? |
|------|-----|------|----------|
| **A. Рантайм** | `Dsl.Core`, `Dsl.Unity`, ваш хост-скрипт | `Assets/` | да |
| **B. Только редактор** | `Dsl.Tests`, `Dsl.Unity/Editor` (импортёр .sal) | `Assets/` | нет (editor-only asmdef) |
| **C. Инструменты** | `Tools/DslCheck`, `Tools/vscode-salamander`, `Examples/DemoHost`, `modkit/` | ВНЕ `Assets/` | нет |

Почему так: `Dsl.Core` собран с `noEngineReferences` (не тянет UnityEngine),
`Dsl.Unity` — тонкий адаптер поверх него. Всё из зоны C — это либо отдельные
консольные `.csproj` с собственным `Main`, либо TypeScript; если занести их
в `Assets`, Unity попытается их скомпилировать (конфликт точек входа, лишний
код в билде). Поэтому зону C держим строго вне `Assets`.

## Шаги интеграции (зона A)

1. Установите Newtonsoft: Package Manager → Add package by name →
   `com.unity.nuget.newtonsoft-json`.
2. Скопируйте в проект **как есть, не переименовывая** (имя сборки задаёт её
   `.asmdef`, а не родительская папка — держите папки с их именами):
   ```
   Assets/Salamander/Dsl.Core/    ← папка Dsl.Core целиком (asmdef Dsl.Core)
   Assets/Salamander/Dsl.Unity/   ← папка Dsl.Unity целиком (asmdef Dsl.Unity),
                                     включая Editor/ с импортёром .sal
   ```
   (по желанию) `Assets/Salamander/Dsl.Tests/` ← `Dsl.Tests` — запускается в Test
   Runner, в билд не входит.

   `Dsl.Unity/Editor/` содержит `SalamanderImporter` (asmdef `Dsl.Unity.Editor`,
   editor-only): он импортирует `.sal` как `TextAsset`. Без него `.sal` нельзя
   перетащить в `_embeddedModules` сцены или загрузить как ассет. В плеер-билд
   импортёр не входит.
3. Напишите свой хост — наследник `ScriptHostBootstrap`, зарегистрируйте API
   игры в `ConfigureHost` через fluent `HostBuilder` (полный образец —
   `Examples/UnityRpg/RpgSimulation.cs`). Важное правило: `ConfigureHost`
   должен только РЕГИСТРИРОВАТЬ, ничего не исполняя, — он вызывается и при
   экспорте манифеста вне Play.
4. Задайте, откуда грузятся моды. По умолчанию бутстрап читает
   `StreamingAssets/<modsFolder>` (поле в инспекторе). Для пользовательских
   модов можно указать любую папку — переопределите `ModsPath`.
5. Для сетевой игры (FishNet и т.п.) переопределите
   `IsAuthority => netObject.IsServerInitialized` — скрипты пойдут только на
   сервере.

Ничего из зоны C в `Assets` не копируется. Всё.

## Экспорт манифеста API (для модеров)

Модерам нужен `salamander-api.json` — слепок того, что игра открыла скриптам.
Источник истины один — ваш `HostRegistry`, поэтому манифест не расходится
с игрой. Два способа выгрузить:

- **Из редактора без Play:** ПКМ по компоненту-хосту в инспекторе →
  «Export API manifest». Файл ляжет в папку модов.
- **Автоматически при Play:** в редакторе бутстрап выгружает манифест при
  старте сцены (тумблер «Export API manifest» в инспекторе).

В билде экспорт не выполняется (обёрнут в `#if UNITY_EDITOR`).

## Сборка и раздача модкита

`modkit/` — это готовая среда для модеров (VS Code + чекер + пример + доки).
Чтобы собрать её под свою игру:

1. Соберите CLI-чекер как самодостаточный бинарник (модеру не нужен .NET):
   ```bash
   dotnet publish Tools/DslCheck -c Release -r win-x64 \
       --self-contained -p:PublishSingleFile=true -o modkit/bin
   ```
   (повторите для `linux-x64` / `osx-x64` по надобности).
2. Положите ваш `salamander-api.json` в `modkit/mods/`.
3. Соберите расширение VS Code в `.vsix`:
   ```bash
   cd Tools/vscode-salamander && npm install && npm run compile && npx @vscode/vsce package
   ```
   Приложите `.vsix` к модкиту (или опубликуйте в маркетплейс).
4. Заархивируйте `modkit/` и раздайте. Модеру достаточно распаковать,
   открыть папку в VS Code, поставить `.vsix` — и писать моды (см.
   `modkit/README.md`).

## Режимы исполнения модулей (кооперативный / синхронный)

Каждый модуль в `module.json` объявляет `"execution"`:

- `"cooperative"` (по умолчанию) — кооперативные файберы: `wait N`, `wait until`,
  `spawn` работают. Подходит для стратегий, RPG, всего, где нужна логика во
  времени и параллельные цепочки.
- `"synchronous"` — без файберов: обработчик события исполняется целиком до
  конца в один проход, `wait`/`wait until`/`spawn` — ошибка компиляции (E0170).
  Подходит для карточных и пошаговых игр: одно событие → весь эффект
  разрешился по порядку, без отложенных продолжений.

Правило целостности: **синхронный модуль может зависеть только от синхронных**
(E0306). Иначе `func` из кооперативной зависимости могла бы сделать `wait` и
приостановить синхронный обработчик — гарантия «до конца» протекла бы.

Режим — свойство модуля, но на практике игра однородна: карточная — все модули
`synchronous`, стратегия — все `cooperative`. Движку менять ничего не нужно:
без `wait`/`spawn` обработчик и так исполняется за один проход, а бюджет/
статистика работают одинаково в обоих режимах. Чекер и modkit понимают поле
автоматически (это тот же компилятор).

## Скрипты, привязанные к карте (как в W3/Arma)

Скрипты можно привязать к конкретной сцене/карте, а не тащить во весь билд.
Движок для этого уже изолирован: `ScriptHostBootstrap` держит собственный
`ScriptEngine` (никакого статического состояния между картами нет), поэтому
всё живёт и умирает вместе со своим GameObject/сценой.

Два способа:

1. **Своя папка на карту (dev/моды).** Поставьте бутстрап в сцену карты и
   укажите `_modsFolder = "Scripts/MapA"`. Скрипты работают, только пока сцена
   загружена. Минус: всё в `StreamingAssets` едет в билд целиком.

2. **Скрипты вшиты в сцену (истинная привязка, как триггеры карты в W3 или
   скрипты миссии в Arma).** В бутстрапе карты заполните `_embeddedModules`:
   каждый модуль — это `module.json` и его `.sal`-файлы как **TextAsset-ы**,
   перетащенные в инспектор (`.sal` распознаётся как TextAsset благодаря
   Editor-импортёру из `Dsl.Unity/Editor`); снимите галку `_loadModsFolder`.
   Такие ассеты попадают в билд **только вместе со сценой, которая на них
   ссылается** (система зависимостей Unity), и компилируются только при
   загрузке этой сцены. Другие карты их не видят и в свой билд не включают.

Изоляция гарантирована: у каждого бутстрапа свой `ScriptEngine`, реестр API,
пул файберов и таблицы — при выгрузке сцены всё уничтожается, файберы
останавливаются. Можно держать одновременно глобальный бутстрап (общие
правила) и пер-карточный (скрипты карты) — события поднимаются на нужный
движок независимо.

Регистрация API (`ConfigureHost`) при этом общая для игры — модель мира одна;
различается только НАБОР скриптов, привязанный к карте.

## Скрипты, привязанные к карте — примечание о хот-релоаде

Хот-релоад с диска работает для папки (`_loadModsFolder`), но не для вшитых
TextAsset-модулей: чтобы подхватить правку вшитого скрипта, нужно
переимпортировать ассет (Unity сам это делает при сохранении в проекте, а в
рантайме билда вшитые скрипты фиксированы). Для итеративной разработки карты
удобнее временно грузить те же файлы из папки, а перед релизом перенести их
в `_embeddedModules`.

## Про размер билда: почему компилятор всё же входит

Сейчас моды компилируются из исходников `.sal` прямо в игре (в т.ч. хот-
релоад), поэтому компилятор (`Dsl.Core` целиком) должен быть в билде — это
норма для игры с модами на стороне игрока.

Если ваша игра НЕ поддерживает пользовательские моды в релизе, а весь контент
авторится на этапе разработки, компилятор в билде не нужен. Чтобы его убрать,
понадобится сериализация скомпилированной программы (грузить готовый байткод
вместо исходников) — это отдельная возможность, в v1 её нет. Заготовка под
неё есть: `CompiledProgram` — чистые данные без «живых» ссылок, а фронтенд
(лексер/парсер/чекер/кодоген) и рантайм (VM/ScriptEngine) уже не зависят друг
от друга по вызовам. Когда появится формат байткода, `Dsl.Core` можно будет
разбить на `Core.Runtime` (в билд) и `Core.Compiler` (только редактор/тулинг)
без переписывания.
